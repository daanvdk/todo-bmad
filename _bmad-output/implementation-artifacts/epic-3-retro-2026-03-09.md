## What Went Well

1. **Complete delivery with zero incidents** — 3/3 stories, all remaining FRs and NFRs, clean codebase.

2. **Playwright MCP had genuine impact** — Being able to take screenshots mid-implementation and verify visual states directly, rather than relying on description, meaningfully accelerated the UX polish and accessibility verification work in Story 3.3. Daan identified this as the standout win of the epic.

3. **Architecture paid off** — Much of Epic 3 was already structurally in place from Epics 1–2: ThemeProvider, ErrorBanner, semantic HTML, aria-labels. Stories were completing a well-sketched picture rather than building from scratch.

4. **Test suite depth** — 72 E2E tests including real WCAG contrast ratio verification (oklch→sRGB→luminance computed in Playwright), multi-browser coverage, and viewport-specific assertions. The SkeletonRow using TodoRow directly made layout shift impossible by construction — architectural quality, not just tested against.

5. **100% follow-through on Epic 2 action items** — Playwright MCP evaluated and enabled, CLAUDE.md sections added, utility file testing working agreement codified. Three for three, matching Epic 1's follow-through rate.

## Challenges

1. **Multiple code review rounds per story** — Story 3.2 had 3 passes + a CI fix; Story 3.3 had 5+ iterations. This combined two distinct issues that are worth separating.

2. **Accessibility gaps consistently caught at review, not implementation** — Checkbox touch target (3.2: 16px instead of 44px), checkbox border contrast (3.3: WCAG 1.4.11 failure), Input.tsx missing focus ring (3.3). The review loop caught all of them, but they recurred in the first pass of each story.

3. **Test infrastructure gotchas** — Each story introduced at least one non-obvious constraint: 15-second E2E timeout for TanStack Query retries (3.1), `{ exact: true }` locator for delete button due to TodoRow computed accessible name (3.2), WebKit Tab navigation guard for known macOS platform behaviour (3.3). All documented and resolved.

4. **Spec gap in Story 3.1** — The skeleton loading spec optimised for row height consistency but missed that 3 skeleton rows without section labels creates a structural layout shift when real content (with "Active"/"Completed" labels) loads. Partially achieves the "no layout shift" AC.

---

## Key Insights

1. **Agentic AI excels at well-specified, testable work. Iterative visual polish based on personal taste is better handled directly.** The multiple Story 3.3 iterations were not all accessibility failures — a significant portion was visual refinement (hover consistency, form focus style, icon colors, spacing) based on Daan's preferences after seeing the rendered result. The agentic feedback loop (describe → implement → inspect → adjust) is slower than just making the CSS change yourself for purely subjective polish work.

2. **Design-to-code fidelity gap: framework defaults ≠ design intent.** Color and spacing values were not precisely encoded in the UX spec — the implementation inherited shadcn/ui defaults that diverged from the mockup. The checkbox contrast failure is the clearest example: the design had visually intentional values; the implementation used whatever the framework shipped. For future projects: pin CSS custom property values explicitly in the UX spec rather than relying on framework defaults.

3. **Practical mitigations for design fidelity in future projects:**
   - Include mockup screenshots as inputs to visual stories
   - Use Playwright MCP for mid-implementation visual checks, not just at review
   - Add a short WCAG micro-checklist to story specs for any AC touching interactive elements (44px touch targets, 3:1 contrast, visible focus rings)

4. **The review loop is a reliable safety net, but the floor can be raised on accessibility.** All accessibility gaps were caught before shipping. The opportunity is catching them during implementation rather than review, primarily through the micro-checklist approach above.
