## What Went Well

1. **Clean delivery with minimal mistakes** — Both stories delivered on the first pass with only small developer refinements needed. No major rework or architectural mistakes.

2. **Architecture spec quality translated directly into implementation** — Detailed planning artifacts (architecture.md, epics.md) reduced ambiguity and kept implementation on track.

3. **Story documentation discipline is paying off** — Dev notes, debug logs, and completion notes in each story file are creating compounding context that transfers cleanly between stories. Daan noted this explicitly as a win.

4. **Code review loop worked** — Story 1.1 code review caught real issues before they became debt: ESLint conflict with Biome, missing backend healthcheck, vitest globals types, missing orval prettier dependency.

5. **Docker Bake restructuring was a deliberate win** — Daan restructured the Docker setup in Story 1.2 to support GHA Docker Bake caching. This also baked the Caddyfile into the proxy image and reduced production volumes to data-only. Future-proofing benefits: custom postgres extensions via `db/Dockerfile`, deploy-by-pulling-containers with no code mounts.

6. **CI orval-freshness job** — A clever quality gate that will prevent API client drift silently. Active from day one.

---

## Challenges

1. **Tooling version nuances require hands-on debugging** — Biome 2.4.5 `organizeImports` config was in the wrong location (caught in code review), Ruff import ordering in `alembic/env.py` needed fixing, Docker Bake compatibility required unplanned Dockerfile additions. None were blockers but each required debugging time.

2. **AI dev agent lacks a verification habit** — Changes were made but not always verified as solving the actual issue. Daan noted friction from making refinement passes where Claude would change something but not confirm the change worked. This is a systemic gap in the dev agent workflow.

---

## Key Insights

1. **"Discovered in practice" improvements are healthy** — The Docker Bake restructuring wasn't in the story spec; it emerged from real CI implementation work. This kind of discovery should be embraced, not treated as scope creep.

2. **Planning artifact quality is the force multiplier** — Epic 2 will benefit from the same level of architectural upfront detail that made Epic 1 smooth.

3. **Meta-tooling work belongs in config, not the sprint backlog** — Working agreements and tooling setup are not stories. They live in CLAUDE.md and settings files.

4. **Iterative is right for E2E test isolation** — Don't pre-solve problems that haven't manifested yet. Address E2E database isolation when it causes actual failures.
