## What Went Well

1. **Smooth execution across both epics** — 7/7 stories delivered with no blockers. Infrastructure and QA epics were treated with the same rigor as feature epics.

2. **Retroactive coverage work found a real bug** — Story 5.1's coverage push discovered a `useRef` vs `let` bug in `useTodoMutations.ts` that had been hiding since Epic 2. The bug was introduced during a code review fix in story 2-4 and survived two epics undetected. This alone justified the coverage investment.

3. **Spec flexibility enabled exceeding requirements** — The coverage ACs specified '>=70%', which gave room to push enforcement to 90% thresholds (actual: 98% BE / 97% FE). Specs that define floors rather than ceilings age better.

4. **Clean security audit** — Zero CVEs across all dependencies, no secrets in git history, CORS properly locked, all containers non-root, all DB access via ORM.

5. **Quality evidence is documented** — Performance benchmarks validated with automated Playwright tests, accessibility score 100/100, security findings report produced. Not just "trust me" — verifiable evidence.

## Challenges

1. **Review-stage fixes bypass review** — The `nextTempId` bug was the poster child. Story 2-4's code review correctly identified a collision risk, but the fix (refactoring to a decrementing counter) was applied during the review stage itself. This introduced a React lifecycle bug (`let` instead of `useRef`) that was never reviewed by fresh eyes. The bug survived through stories 3-1 to 4-3 because optimistic IDs are short-lived and no unit test existed for the hook.

2. **Axe contrast enforcement too rigid** — The tool doesn't differentiate between primary body text and deliberately de-emphasized secondary text (timestamps). WCAG AA requires 4.5:1 for all normal text regardless of visual hierarchy intent. Many industry-standard design systems (e.g., Material Design) intentionally deviate from this. Axe offers no mechanism for per-element rule exceptions — you can exclude elements from the entire scan or disable a rule globally, but you can't say "check everything except contrast on these elements."

3. **Minor tooling friction** — `uv audit` was experimental and didn't return CVE results (fell back to `pip-audit`). `pip-audit` crashed on Python 3.13. Axe miscomputed contrast ratios with oklch CSS values in Firefox/WebKit, requiring conversion to hex. These were ecosystem issues, not project issues.

---

## Key Insights

1. **If a review fix needs its own test, it's too big for the review stage.** The safest default is: review never fixes code, always sends findings back to dev. This prevents the class of escaped bugs where review-stage changes bypass review entirely.

2. **Axe is valuable for structural accessibility (ARIA, keyboard nav, focus management, semantic HTML) but contrast enforcement should be disabled.** Contrast is a design decision best handled in design review, not as an automated gate. Running axe with the `color-contrast` rule disabled captures 90%+ of its value without the false-positive friction.

3. **Specs with floor thresholds enable teams to exceed expectations.** '>=70%' allowed pushing to 90% enforcement. '>=90 Lighthouse score' gave room for the team to achieve 100. Exact targets would have been ceilings instead of floors.

4. **Report test counts honestly.** 32 E2E tests running across 3 browsers is 32 tests, not 96. The browser multiplication inflates the number and obscures actual test diversity.

5. **BMAD workflows are designed to be adapted locally.** Customizing the review workflow to formalize the "fix in review vs. send back to dev" boundary is encouraged, not "impure."
