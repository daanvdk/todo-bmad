## What Went Well

1. **Complete delivery with zero incidents** — 4/4 stories, 9/9 FRs, clean codebase. The full task management loop works.

2. **Optimistic update pattern executed consistently and evolved well** — Pattern applied correctly in Story 2.3, repeated in Story 2.4, then proactively extracted into `makeOptimisticHandlers` generic helper. Organic architecture improvement without prompting.

3. **Code review loop caught real issues every story** — Not style nits, but correctness issues: deprecated type imports, missing `order_by` for deterministic results, invalid HTML nesting, naming inconsistencies, missing test coverage. The process is working exactly as intended.

4. **Working agreements created a quality flywheel** — CLAUDE.md verification habits, docs-in-sync practice, and code review discipline compounded across all four stories. Each story benefited from the improved context and habits of the previous one. Daan identified this explicitly as the standout win of the epic.

5. **38 unit tests + 9 E2E tests** — Real coverage that will protect Epic 3 refactoring and additions.

6. **100% follow-through on Epic 1 action items** — Including a deliberate, well-reasoned decision on the PostToolUse hooks investigation (investigated, decided against automation, incorporated into habits instead).

## Challenges

1. **New utility file shipped without tests (Story 2.4)** — `optimisticMutation.ts` introduced a non-trivial generic helper (`makeOptimisticHandlers`) with no unit tests in the initial pass. Caught in code review and fixed, but this is the one pattern to close proactively going forward.

2. **Edge case test coverage discovered at review** — Across most stories, review added missing test cases: bidirectional PATCH test (2.1), AC7 coverage (2.2), refocus test and integration test (2.3), `optimisticMutation.test.ts` (2.4). The review safety net caught these, but the floor can be raised.

3. **Story spec had a stale assumption (Story 2.4)** — The spec said `ErrorBanner` already existed; it didn't. Agent handled it gracefully by creating it, but it's a story preparation quality note.

4. **Story 2.2 required two code review rounds** — The only story in the epic that needed a second review pass. Scope was large (bootstrapping entire frontend UI layer) and HTML validity issues appeared in both production code and test markup.

**Important context:** All challenges were caught and resolved by the existing process. None became actual debt or a production issue. These are opportunities to raise the floor, not signs of systemic failure.

---

## Key Insights

1. **The review loop is a reliable safety net — but utility file testing is the one gap to close proactively.** A pure helper function with no side effects is actually the easiest thing to unit test. There's no excuse for shipping `lib/*.ts` files without tests.

2. **Working agreement documentation compounds.** The investment in CLAUDE.md and story spec quality pays dividends on every subsequent story. The more precisely the agent understands expectations, the less cleanup is needed at review.

3. **Playwright MCP will meaningfully extend agent capability for Epic 3.** Stories 3.1-3.3 involve visual states (loading skeletons, error banner, theme toggle), accessibility verification (keyboard nav, focus rings), and responsive layout — all things that benefit from a running browser rather than just test assertions.
